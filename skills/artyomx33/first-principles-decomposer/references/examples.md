# 第一性原理案例 - Artem 的实践

## 案例一：TeddySnaps 照片系统

### 表面问题
"我们需要给家长发送更多照片"

### 被挑战的假设
1. ❌ 家长想要更多照片 → 实际上：他们想要正确的照片
2. ❌ 所有时刻都平等 → 实际上：有他们孩子的时刻才重要
3. ❌ 员工应该筛选 → 实际上：员工时间昂贵且不一致

### 根本真理
- 家长想要证明他们特定的孩子安全/快乐
- 脸 = 身份 = "那是我的孩子"
- 时间 = 上下文 = "这发生在今天"
- 自动 > 手动（为了一致性）

### 重建的解决方案
AI 人脸识别 + 自动标记 + 家长特定动态
员工自由拍照 → 系统自动排序和交付

## 案例二：TISA 注册流程

### 表面问题
"我们的注册流程太慢了"

### 被挑战的假设
1. ❌ 我们需要提前获取所有信息 → 实际上：我们需要足够信息来做决策
2. ❌ 表单必须全面 → 实际上：表单创造摩擦
3. ❌ 家长想要提供信息 → 实际上：家长想要收到确认

### 根本真理
- 注册 = 承诺交换（家长承诺孩子，学校承诺学位）
- 最低可行信息：孩子姓名、年龄、家长联系方式
- 其他都可以 later
- 响应速度 = 质量感知

### 重建的解决方案
第一阶段：姓名 + 年龄 + 电子邮件（30秒）→ 即时学位保留
第二阶段：详细表单（承诺之后）
结果：速度快10倍，转化率更高

## 案例三：GolfTab 点餐

### 表面问题
"高尔夫球手需要在球场上点餐"

### 被挑战的假设
1. ❌ 需要完整餐厅菜单 → 实际上：简化菜单效果更好
2. ❌ 需要精确送达时间 → 实际上，"在X洞之前"就足够了
3. ❌ 复杂付款流程 → 实际上：房间/记帐收费，稍后结算

### 根本真理
- 高尔夫球手的手经常脏/戴手套
- 球场上的注意力跨度 = 极小
- 位置 = 洞号（已通过发球时间知道）
- 速度 > 选项

### 重建的解决方案
5次点击点餐：项目 → 数量 → 洞 → 确认 → 完成
无需浏览菜单，无需付款摩擦，无需输入地址

## 案例四：员工排班（TeddyKids）

### 表面问题
"员工排班复杂且耗时"

### 被挑战的假设
1. ❌ 经理必须创建排班 → 实际上：AI可以起草，经理批准
2. ❌ 员工偏好不可预测 → 实际上：模式随时间出现
3. ❌ 比例合规是手动的 → 实际上：这是纯数学

### 根本真理
- 孩子：员工比例是法律规定的（确定性的）
- 员工可用性有模式（可学习的）
- 成本优化是数学问题
- 人力监督仅用于例外情况

### 重建的解决方案
算法生成排班 → 经理审核例外 → 员工确认
80%自动化，20%人力判断
